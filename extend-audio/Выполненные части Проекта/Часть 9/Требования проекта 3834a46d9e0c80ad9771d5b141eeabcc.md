# Требования проекта

# Для чего проекту требования

Требования нужны для того, чтобы проект перестал быть набором мыслей и превратился в понятное описание будущей системы. По требованиям можно проверить, что именно должно быть реализовано и как понять, что это действительно работает. Требования связывают MVP, дизайн экранов, базу данных и критерии приёмки в единую систему.

## Ограничения проекта

| Ограничение | Описание | Как влияет |
| --- | --- | --- |
| Платформа | Первая версия разрабатывается под Android | Не нужно проектировать версии для iOS/Windows сразу |
| Сроки | Проект должен быть готов к дате защиты | В MVP остаются только обязательные функции |
| Данные | Данные хранятся локально на устройстве | Синхронизация между устройствами не входит в MVP |
| Команда | Проект выполняется индивидуально | Объём MVP ограничен одним разработчиком |
| Интернет | Приложение работает без подключения к сети | Не нужны API, сервер, авторизация |
| Технологии | Android Studio, Kotlin/Java, локальная БД (Room/SQLite) | Используются изученные инструменты |

## Критерии приёмки» для Must Have требований

| ID | Критерии приёмки |
| --- | --- |
| FR-01 | 1) Кнопка «Загрузить» открывает файловый менеджер.2) Выбранный аудиофайл добавляется в библиотеку.3) Форматы: MP3, FLAC, WAV. |
| FR-02 | 1) Нажатие на трек запускает воспроизведение.2) Звук слышен через наушники/динамик.3) Кнопки play/pause работают. |

| FR-05 | 1) Кнопка «Сохранить пресет» доступна после настройки эквалайзера.2) После нажатия пресет появляется в локальном списке.3) Пресет содержит название и значения частот. |

## Итоговый вывод по требованиям

Функциональные требования EXTEND Audio описывают ключевые действия пользователя: загрузка трека, настройка эквалайзера, создание и применение пресетов, прослушивание музыки. Все Must Have требования связаны с главным сценарием MVP и проверяются конкретными действиями. Нефункциональные требования обеспечивают надёжность хранения пресетов, скорость открытия экранов и удобство управления плеером. Ограничения проекта фиксируют рамки: Android, локальные данные, один разработчик, приложение без доступа к интернету. Критерии приёмки позволяют проверить каждое требование на защите. Такая структура требований связывает MVP из части 7, модель разработки из части 8 и будущее проектирование экранов и данных.
