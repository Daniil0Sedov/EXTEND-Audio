# Выбор модели разработки ПО

## Коротко обо всех моделях

| Модель | Коротко | Когда особенно подходит |
| --- | --- | --- |
| Каскадная модель (Waterfall) | Последовательные этапы: требования, проектирование, разработка, тестирование, внедрение. | Когда требования заранее понятны, стабильны и почти не будут меняться. |
| V-образная модель | Похожа на Waterfall, но каждому этапу проектирования соответствует уровень тестирования. | Когда важна проверяемость требований, качество и надёжность. |
| Итеративная модель | Один и тот же продукт постепенно уточняется через повторяющиеся циклы. | Когда нужно несколько раз дорабатывать решение, пока оно не станет точным. |
| Инкрементная модель | Продукт собирается частями: каждый инкремент добавляет рабочий кусок функциональности. | Когда проект можно разделить на модули и быстро показать MVP. |
| Спиральная модель | Разработка идёт витками, на каждом витке отдельно анализируются риски. | Когда проект крупный, дорогой, рискованный и цена ошибки высокая. |
| Agile / Scrum | Работа короткими спринтами, постоянная обратная связь и гибкое изменение плана. | Когда требования могут меняться, а пользователи должны регулярно видеть результат. |
| RAD | Быстрая разработка через готовые компоненты, конструкторы, шаблоны и частые демонстрации. | Когда нужно быстро собрать типовое приложение из готовых решений. |
| Прототипная модель | Сначала создаётся прототип, чтобы проверить идею, интерфейс и ожидания пользователя. | Когда непонятен UX, сценарий или точный вид будущего продукта. |
| DevOps | Непрерывная разработка, автоматизация сборки, тестирования, публикации и мониторинга. | Когда продукт нужно постоянно обновлять и поддерживать после релиза. |

# Выбранная модель для EXTEND Audio

Основная модель: Agile / Scrum с элементами прототипной модели и инкрементной поставки.

## Почему так? Основные аргументы

1. Требования могут уточняться. По результатам анкетирования (15 человек) выяснилось, что гипотеза подтверждена частично — есть уточнения по функционалу. Это значит, что фиксировать всё заранее и идти по обычному Waterfall нельзя.
2. Интерфейс важно проверять на пользователя. Экраны сгенерированы через Google Stitch, но до реальной разработки непонятно, насколько они удобны. Agile позволяет показывать промежуточные результаты и корректировать.
3. Проект легко делится на функции. Поиск пресетов, эквалайзер, плеер, библиотека треков — в целом это независимые модули, которые можно добавлять по очереди.
4. Один разработчик. Agile/Scrum в упрощённом виде (без команды) позволит гибко расставлять приоритеты и не тратить время на формальную документацию, которая не нужна для проекта такого рода.

## Почему не эдак?

| Модель | Почему не подходит |
| --- | --- |
| Waterfall | Требования ещё не полностью стабильны. После анкетирования могут появиться уточнения. A Waterfall предполагает, что всё известно до начала разработки. У меня не гос проект |
| V-модель | Проект не является критичной системой. V-модель слишком формальна для моего простого аудиоплеера. |
| Спиральная | Риски есть, но они не настолько велики, чтобы строить работу вокруг сложного управления рисками. Слишком много для одного человека. |
| RAD | Проект не типовой — нельзя просто собрать из готовых шаблонов. Нужна своя логика эквалайзера и поиска пресетов. |
| Итеративная | Близка к Agile, но сама по себе не описывает организацию работы (спринты, приоритеты). Лучше использовать Agile как основу. |

## Примерная схема разработки проекта

[Этап 1. Прототипирование]
Экраны из Google Stitch → проверка UX → фидбек от 2.5 человека
Результат: утверждённые экраны
↓
[Этап 2. Инкремент #1 — Ядро плеера]
Загрузка трека → прослушивание → базовый UI
Результат: рабочий плеер сначала без эквалайзера
↓
[Этап 3. Инкремент #2 — Эквалайзер]
Ползунки частот → применение к звуку → применение пресета
Результат: эквалайзер работает
↓
[Этап 4. Инкремент #3 — Пресеты]
Создание пресета → сохранение в локальную БД → выбор из списка
Результат: пользователь может создавать и выбирать пресеты
↓
[Этап 5. Инкремент #4 — Библиотека]
Библиотека треков → навигация → загрузка
Результат: полный основной сценарий MVP
↓
[Этап 6. Тестирование и доработка]
Ручное тестирование → исправление ошибок → финальная сборка
Результат: готовый MVP
↓
[Проверка: пользователь прошёл сценарий?]
Да → защита / Нет → доработка

## Итоговый вывод по выбору модели

Для разработки EXTEND Audio выбрана модель Agile / Scrum с элементами прототипной и инкрементной моделей. Это позволяет начать с проверки экранов, а затем постепенно добавлять функции: плеер, эквалайзер, пресеты, библиотека. Каждый инкремент — это рабочая часть продукта, которую можно показать и проверить. Другие модели (Waterfall, V-модель, спиральная) слишком формальны или тяжелы для учебного проекта с одним разработчиком. RAD не подходит, потому что проект не типовой и требует собственной логики. Такой подход позволяет довести MVP до готового состояния за 3-4 спринта и иметь рабочий результат на каждом этапе.
