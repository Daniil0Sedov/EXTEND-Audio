## **Цель занятия**

#### На предыдущих этапах были подготовлены требования, техническое задание, дизайн, пользовательские сценарии, база данных и архитектура приложения. Теперь студент начинает непосредственную разработку мобильного приложения в Android Studio.

#### Цель этой части — не просто «написать код», а организовать разработку так, как это происходит в реальном проекте: создать репозиторий, подключить его к Android Studio, загрузить необходимые ресурсы, вести осмысленную историю коммитов и документировать ход работы в Notion / Buildin.

| После выполнения части 15 у студента должно быть — созданное мобильное приложение в Android Studio; — репозиторий GitHub с понятной историей разработки; — структура проекта, связанная с архитектурой из части 14; — подключенные изображения, иконки и другие ресурсы; — страница в Notion / Buildin с этапами разработки, ссылками, скриншотами и пояснениями; — понимание, где в коде находится экран, навигация, модель данных, логика и ресурсы. |

## **Что значит «разработка приложения»**

#### Разработка приложения — это последовательное превращение проектной документации в работающий продукт. На этом этапе студент берет дизайн, User Flow, ERD, архитектуру и начинает переносить их в код.

#### Важно понимать: разработка не начинается с хаотичного создания экранов. Сначала нужно подготовить проект, структуру, репозиторий, ресурсы и план реализации. Только после этого следует переходить к написанию экранов, логики и подключению данных.

| Что уже есть | Во что превращается в разработке |
|---|---|
| Требования и ТЗ | список функций приложения и критерии готовности |
| Дизайн экранов | UI-экраны, компоненты, цвета, иконки, отступы |
| User Flow | навигация между экранами и сценарии пользователя |
| ERD / UML | модели данных, классы, таблицы или локальное хранилище |
| Архитектура | структура папок, слои приложения, разделение ответственности |

## **Подготовка рабочего проекта**

#### Перед началом кодирования студент должен создать или открыть проект в Android Studio и привести его к рабочему состоянию. Проект должен запускаться, собираться без критических ошибок и быть готовым к подключению Git.

### **Минимальная подготовка**

#### • создать проект Android Studio с понятным названием;

#### • проверить, что проект запускается на эмуляторе или физическом устройстве;

#### • настроить структуру папок в соответствии с выбранной архитектурой;

#### • удалить лишний демонстрационный код, который не относится к проекту;

#### • подготовить папки для экранов, моделей, навигации, репозиториев и ресурсов;

#### • зафиксировать первый рабочий запуск скриншотом в Notion / Buildin.

| Не принимается как полноценная разработка — проект без запуска; — проект без репозитория; — один огромный файл со всем кодом; — отсутствие скриншотов процесса; — отсутствие объяснения, где находится какая часть приложения. |
|---|

## **Репозиторий GitHub и связь с Android Studio**

#### Каждый студент должен создать отдельный репозиторий GitHub для своего итогового проекта. Репозиторий нужен не только для хранения кода, но и для проверки процесса разработки: преподаватель должен видеть, как проект создавался по шагам.

### **Что нужно сделать**

#### • создать репозиторий на GitHub с понятным названием, например student-helper-app или habit-tracker-mobile;

#### • добавить README.md с кратким описанием проекта;

#### • связать локальный проект Android Studio с Git;

#### • сделать первый коммит после создания проекта;

#### • отправить проект на GitHub через push;

#### • прикрепить ссылку на репозиторий в Notion / Buildin.

## **Зачем нужны коммиты**

#### Коммит — это сохраненная точка в истории разработки. По коммитам видно, что студент действительно работал над проектом постепенно, а не загрузил готовый архив в последний день.

#### Хорошая история коммитов помогает понять, когда был создан экран, когда добавлена навигация, когда подключены изображения, когда исправлена ошибка и когда подготовлена финальная версия.

| Требование по количеству коммитов — Минимум: 12 осмысленных коммитов за весь проект. — Хороший уровень: 18–25 коммитов. — Коммиты должны отражать реальные этапы разработки, а не быть набором сообщений «update», «final», «готово». |
|---|

## **Как правильно называть коммиты**

#### Название коммита должно коротко объяснять, что именно изменилось. Желательно использовать единый стиль. Можно использовать английские названия, потому что это распространенная практика в разработке.

| Тип | Когда использовать | Пример |
|---|---|---|
| init | первое создание проекта | init: create android project |
| docs | изменения в README, документации, описании | docs: add project description |
| feat | новая функция, экран, сценарий | feat: add login screen |
| ui | визуальные правки интерфейса | ui: improve profile screen layout |
| fix | исправление ошибки | fix: handle empty login fields |
| refactor | улучшение структуры кода без изменения поведения | refactor: split auth logic |
| data | модели, база данных, хранилище | data: add task model |
| test | проверка, тестовые данные, чек-листы | test: add manual test checklist |

| Плохо | Почему плохо | Лучше |
|---|---|---|
| update | непонятно, что изменилось | feat: add home screen |
| final | не отражает действие | docs: prepare final README |
| исправил | непонятно, что исправлено | fix: correct navigation to profile |
| работа | неинформативно | ui: add onboarding illustrations |

## **Какие изображения и ресурсы нужно подготовить**

#### Перед разработкой студент должен собрать все визуальные ресурсы, которые будут использоваться в приложении. Это нужно сделать заранее, чтобы не искать картинки хаотично в процессе кодирования.

### **Обязательный набор ресурсов**

| Ресурс | Для чего нужен | Пример названия файла |
|---|---|---|
| Логотип приложения | иконка, splash screen, экран входа | logo_app.png |
| Иконка приложения | отображение на устройстве | ic_launcher.png |
| Иконки интерфейса | нижнее меню, кнопки, профиль, настройки | icon_profile.svg |
| Иллюстрации пустых состояний | если нет задач, курсов, заметок, данных | empty_tasks.png |
| Изображения onboarding | первые экраны знакомства с приложением | onboarding_1.png |
| Заглушки / аватары | тестовые пользователи, карточки | avatar_default.png |
| Скриншоты дизайна | сверка реализации с макетом | design_home.png |

| Правило именования ресурсов — Названия должны быть понятными и без пробелов: icon_profile.svg, empty_courses.png, logo_app.png. — Не использовать имена image1.png, finalfinal.png, картинка.png. — Если ресурс создан нейросетью, нужно указать промт или источник. |
|---|

## **Структура проекта и понимание кода**

#### Студент должен не только написать код, но и показать, что понимает, где находится каждая часть приложения. Для этого в Notion / Buildin нужно создать карту проекта: какие папки и файлы за что отвечают.

### **Что нужно объяснить по своему проекту**

#### • где находится главный экран приложения;

#### • где находится экран входа или регистрации, если он есть;

#### • где описана навигация между экранами;

#### • где находятся модели данных;

#### • где расположена логика работы с хранилищем, API или локальными данными;

#### • где подключены изображения, иконки, цвета и строки;

#### • какой коммит добавил или изменил каждую важную часть.

## **Рекомендуемая структура Android-проекта**

#### Структура может отличаться в зависимости от выбранной технологии и архитектуры, но студент должен придерживаться понятного разделения ответственности. Нельзя складывать весь код в один файл или одну папку без логики.

| Папка / слой | Что хранится | Пример |
|---|---|---|
| ui / screens | экраны приложения | LoginScreen, HomeScreen, ProfileScreen |
| ui / components | переиспользуемые элементы интерфейса | ProjectCard, BottomMenu, AppButton |
| navigation | маршруты и переходы | AppNavGraph, routes |
| model / data | модели данных | User, Task, Course, Project |
| repository | получение и сохранение данных | TaskRepository |
| database / storage | локальная база, DAO, настройки | AppDatabase, TaskDao |
| theme / resources | цвета, стили, изображения | Color, Theme, drawable |

| Для простого проекта допустимо упростить структуру — Если приложение небольшое, не нужно искусственно усложнять архитектуру. — Но даже в простом проекте должны быть отделены экраны, модели данных, навигация и ресурсы. — Студент должен уметь объяснить выбранную структуру. |
|---|

## **Последовательность разработки MVP**

#### Работу нужно выполнять по этапам. Каждый этап желательно фиксировать отдельным коммитом и короткой записью в Notion / Buildin.

| Этап | Что сделать | Что зафиксировать |
|---|---|---|
| 1. Создание проекта | создать Android-проект и проверить запуск | скриншот запуска + init-коммит |
| 2. README | описать цель проекта, функции, стек | docs-коммит + ссылка на GitHub |
| 3. Ресурсы | добавить логотип, иконки, иллюстрации | скрин папки ресурсов |
| 4. Базовые экраны | создать главные экраны из дизайна | скриншоты экранов |
| 5. Навигация | связать экраны по User Flow | схема + commit feat/navigation |
| 6. Модели данных | создать классы/модели по ERD/UML | скрин структуры кода |
| 7. Логика | реализовать основной сценарий MVP | видео/скрин прохождения сценария |
| 8. Хранение данных | подключить локальное хранение или mock/API | описание решения |
| 9. Обработка ошибок | пустые поля, неверный ввод, отсутствие данных | скрин ошибок |
| 10. Финальная проверка | пройти чек-лист MVP | таблица проверки |

## **Что оформлять в Notion / Buildin**

#### Страница разработки нужна для того, чтобы преподаватель видел не только итоговый результат, но и последовательность работы. Это особенно важно, если приложение большое или студент использовал нейросеть.

| Раздел страницы | Что туда поместить |
|---|---|
| 1. Ссылка на GitHub | ссылка на репозиторий, статус доступа, ветка main/master |
| 2. Стек проекта | Android Studio, Kotlin/Java, Jetpack Compose/XML, Room/SQLite/API и т.д. |
| 3. План разработки | таблица этапов: To Do, In Progress, Done |
| 4. Скриншоты процесса | создание проекта, структура папок, ресурсы, экраны |
| 5. Карта кода | таблица: файл/папка — за что отвечает |
| 6. Коммиты | ссылка на историю коммитов + 5–7 ключевых коммитов с пояснениями |
| 7. Ресурсы | какие картинки использованы, где лежат, откуда взяты или каким промтом созданы |
| 8. Проблемы и решения | какие ошибки возникали, как исправлены |
| 9. Демонстрация MVP | скриншоты или короткое видео прохождения основного сценария |
| 10. Самооценка | что готово, что не готово, что улучшить |

## **Использование нейросети при разработке**

#### Нейросеть можно использовать как помощника: для объяснения ошибок, генерации шаблонов, проверки структуры проекта, написания README, подготовки тестовых данных и улучшения UI. Но студент должен понимать результат и уметь объяснить, что было изменено.

### **Что можно просить у нейросети**

#### • помоги разбить приложение на экраны, модели и компоненты;

#### • предложи структуру папок для моего Android-проекта;

#### • проверь название коммитов и предложи более понятные;

#### • объясни ошибку сборки Android Studio простыми словами;

#### • составь README.md по моему проекту;

#### • проверь, соответствует ли код моей архитектуре;

#### • составь чек-лист ручного тестирования MVP.

| Что обязательно фиксировать при работе с ИИ — Если нейросеть помогла написать важный фрагмент кода, студент должен проверить и объяснить его. — Если сгенерированы изображения, нужно сохранить промт или источник. — Если нейросеть предложила архитектуру, нужно сравнить ее с частью 14 и выбрать осознанно. — Нельзя сдавать код, который студент не может объяснить. |

## **Практическое задание**

#### На основе своего итогового проекта начните разработку мобильного приложения в Android Studio и оформите процесс так, чтобы его можно было проверить по GitHub и Notion / Buildin.

### **Задание выполняется по шагам**

#### • Создайте репозиторий GitHub и свяжите его с Android Studio.

#### • Добавьте README.md с названием, целью, функциями MVP и используемым стеком.

#### • Создайте базовую структуру проекта: экраны, навигация, модели, ресурсы, работа с данными.

#### • Реализуйте первые ключевые экраны по дизайну из части 11.

#### • Свяжите экраны навигацией по User Flow из части 12.

#### • Создайте модели данных на основе ERD/UML из части 13.

#### • Начните реализацию основного сценария MVP.

#### • Сделайте не менее 12 осмысленных коммитов.

#### • Оформите страницу разработки в Notion / Buildin.

#### • Подготовьте скриншоты и объяснение, где находится каждая важная часть кода.

## **Что должно быть сдано**
