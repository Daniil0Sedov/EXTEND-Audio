# **Часть 14**

# **Архитектура приложения**

### *Методичка и практическое задание для итогового мобильного проекта*

Цель части: научиться описывать приложение не только как набор экранов, но как систему: какие в ней есть модули, слои, данные, внешние сервисы, потоки и архитектурные решения.

Результат работы: студент оформляет страницу «Архитектура приложения» в Notion / Buildin, создает редактируемые архитектурные схемы и объясняет, почему выбрал именно такую структуру проекта.

## **1. Зачем нужна архитектура приложения**

Архитектура приложения — это способ описать, из каких крупных частей состоит система, как эти части связаны между собой и почему они устроены именно так.

Если требования отвечают на вопрос «что должно быть сделано», а дизайн показывает «как это выглядит для пользователя», то архитектура отвечает на вопрос «как это будет устроено внутри».

**• **помогает не потеряться в проекте, когда экранов и функций становится много;

**• **показывает, где находится интерфейс, бизнес-логика, работа с базой данных и внешними сервисами;

**• **помогает заранее увидеть слабые места: дублирование логики, прямой доступ UI к базе, отсутствие обработки ошибок;

**• **делает проект понятным для преподавателя, команды и будущего разработчика;

**• **помогает связать ТЗ, User Flow, ERD и будущую реализацию в коде.

## **2. Что считать архитектурным решением**

Архитектурное решение — это значимый выбор, который влияет на структуру проекта и который потом сложно менять без последствий.

| Решение | Почему это важно |
|---|---|
| Хранить данные локально или на сервере | Определяет, нужна ли авторизация, синхронизация, API и backend. |
| Делить приложение на слои или писать всю логику в экранах | Влияет на читаемость, тестируемость и возможность доработки проекта. |
| Использовать REST API, Firebase/Supabase или мок-данные | Определяет способ получения и изменения данных. |
| Выделять Repository / Service / UseCase | Показывает, как приложение отделяет интерфейс от работы с данными. |
| Поддерживать offline-режим или нет | Влияет на кэш, локальную базу и обработку конфликтов данных. |

Важный принцип: архитектура не должна быть «сложной ради сложности». Для студенческого MVP лучше выбрать простую, понятную и объяснимую архитектуру.

## **3. Уровни архитектуры**

Архитектуру удобно описывать по уровням. Один уровень показывает систему в целом, другой — приложение внутри, третий — конкретные модули и классы.

| Уровень | Что показывает | Что студент должен оформить |
|---|---|---|
| Системный контекст | Пользователи, приложение, backend, база данных, внешние сервисы. | Схема контекста. |
| Контейнеры | Крупные исполняемые части: мобильное приложение, API, БД, хранилище файлов. | Схема контейнеров. |
| Компоненты | Модули приложения: авторизация, каталог, профиль, задачи, уведомления. | Схема модулей. |
| Код | Классы, функции, модели, репозитории, DTO, SQL. | Структура папок и ключевых классов. |

## **4. Архитектурные варианты**

Для итогового мобильного проекта студент должен выбрать один основной архитектурный вариант. Выбор зависит от того, где хранятся данные, нужен ли сервер, есть ли авторизация и синхронизация.

| Вариант | Когда подходит | Когда не подходит |
|---|---|---|
| Локальное приложение | Личные данные: заметки, задачи, трекер. Данные можно хранить на устройстве. | Нужны общие данные, админка или онлайн-синхронизация. |
| Клиент-сервер | Нужны аккаунты, общий каталог, роли, хранение данных на сервере. | Нет времени делать backend или проект можно решить локально. |
| Гибрид | Часть данных локально, часть с API. Нужен кэш или offline-режим. | Проект простой, а кэш только усложнит реализацию. |
| Модульный монолит | Есть небольшой backend, который проще держать одним приложением. | Части должны независимо масштабироваться. |
| Микросервисы | Большая система с независимыми сервисами и нагрузкой. | Почти всегда лишнее для студенческого MVP. |

## **5. Слоистая архитектура приложения**

Для большинства учебных мобильных проектов лучше всего подходит слоистая архитектура. Она помогает отделить интерфейс от логики и работы с данными.

| Слой | Ответственность | Пример |
|---|---|---|
| UI | Отображает экран и принимает действия пользователя. | Кнопка «Сохранить», список карточек, форма входа. |
| Presentation | Хранит состояние экрана и вызывает сценарии. | ViewModel, Controller, State. |
| Domain | Описывает бизнес-правила и действия приложения. | Создать задачу, проверить права, рассчитать прогресс. |
| Data | Работает с источниками данных. | Repository, API client, Room/SQLite, DTO. |

Антипаттерн: экран напрямую обращается к базе данных или API и сам выполняет все проверки. В маленьком проекте это может работать, но быстро превращается в хаос.

## **6. Клиент-серверная архитектура**

Клиент-серверная архитектура нужна, если мобильное приложение не может хранить все данные только на устройстве пользователя. Например, если есть аккаунты, общий каталог, роли, админка или обмен данными между пользователями.

**• **клиент отвечает за интерфейс, навигацию и отправку запросов;

**• **backend отвечает за бизнес-правила, авторизацию, проверку данных и доступ к базе;

**• **база данных хранит пользователей, записи, заявки, комментарии, статусы и связи;

**• **внешние сервисы могут отвечать за файлы, уведомления, карты, оплату, ИИ или почту.

Для учебного проекта backend можно заменить мок-данными, Firebase, Supabase или простым API. Главное — честно указать выбранный вариант в архитектуре.

## **7. Декомпозиция на модули**

Декомпозиция — это разделение приложения на понятные части. Каждая часть должна иметь свою ответственность. Чем меньше модуль знает о других модулях, тем проще развивать проект.

| Модуль | Что включает | Пример ответственности |
|---|---|---|
| Auth | Вход, регистрация, восстановление доступа. | Проверить данные входа и сохранить токен. |
| Profile | Профиль пользователя, настройки личных данных. | Показать и обновить данные профиля. |
| Catalog | Список сущностей: курсы, товары, задачи, мероприятия. | Загрузить список и применить фильтры. |
| Details | Детальная страница выбранного объекта. | Показать подробности и доступные действия. |
| Notifications | Уведомления, статусы, события. | Показать важные изменения пользователю. |

## **8. Ассоциации между модулями**

Ассоциация в архитектуре — это связь между частями приложения. Например, экран профиля использует данные пользователя, а модуль задач использует идентификатор пользователя для фильтрации записей.

Важно не только нарисовать модули, но и показать, как они взаимодействуют: кто кого вызывает, какие данные передаются, где находится источник правды.

| Тип связи | Пример | Что указать на схеме |
|---|---|---|
| Навигационная | Главный экран открывает экран профиля. | Стрелка перехода между экранами. |
| Данные | Экран списка получает элементы из Repository. | Источник данных и направление передачи. |
| Бизнес-логика | UseCase проверяет возможность выполнения действия. | Какой модуль принимает решение. |
| Внешняя интеграция | Приложение отправляет запрос в API. | Протокол, endpoint или сервис. |
| Событие | После создания записи появляется уведомление. | Событие и получатель события. |

Хорошая схема показывает не просто «соединенные квадраты», а смысл связей между ними.

## **9. Потоки данных**

Поток данных показывает, как действие пользователя превращается в изменение состояния приложения. Это особенно важно для форм, авторизации, создания записей, загрузки списков и обработки ошибок.

**• **начинайте поток с действия пользователя;

**• **показывайте, какой экран или компонент принимает действие;

**• **показывайте, куда уходит запрос: Repository, API, база данных, кэш;

**• **показывайте успешный результат и ошибочный сценарий;

**• **не забывайте про загрузку, пустое состояние и ошибку сети.

## **10. Мини-набор архитектурных схем**

В этой части студенту не нужно создавать профессиональную архитектурную документацию на десятки страниц. Нужен минимальный набор схем, который объясняет устройство проекта.

| Схема | Что на ней должно быть |
|---|---|
| 1. Системный контекст | Пользователь, мобильное приложение, backend/API, база данных, внешние сервисы. |
| 2. Контейнерная схема | Крупные части системы: приложение, сервер, БД, хранилище, сервис уведомлений. |
| 3. Схема модулей | Модули приложения и связи между ними. |
| 4. Схема потоков данных | 1-2 ключевых сценария: вход, создание записи, загрузка списка, отправка заявки. |
| 5. Структура проекта | Папки, основные классы/файлы, назначение директорий. |
| 6. ADR | 2-3 архитектурных решения с обоснованием. |

## **11. ADR: фиксация архитектурных решений**

ADR — Architecture Decision Record. Это короткая запись, где студент объясняет, какое архитектурное решение было принято и почему.

| Раздел ADR | Что написать |
|---|---|
| Контекст | Какая проблема или выбор возникли в проекте. |
| Решение | Что именно выбрано. |
| Альтернативы | Какие варианты рассматривались. |
| Причина выбора | Почему выбран этот вариант. |
| Последствия | Что это упрощает и что может усложнить. |

Пример ADR:

Контекст: в приложении нужно хранить личные задачи пользователя. Решение: использовать локальную базу SQLite/Room. Альтернативы: хранить данные в файле JSON или на сервере. Причина выбора: для MVP не нужна синхронизация между устройствами. Последствия: приложение работает без интернета, но данные не доступны с другого устройства.

## **12. Как выбрать архитектуру**

Архитектуру выбирают не по моде, а по ограничениям проекта. Студент должен уметь объяснить свой выбор через требования, User Flow, ERD и сроки реализации.

| Вопрос | Если ответ «да» | Если ответ «нет» |
|---|---|---|
| Есть ли общие данные для всех пользователей? | Скорее всего нужен backend или облачный сервис. | Можно рассмотреть локальную БД. |
| Нужна ли авторизация? | Нужно продумать хранение токена, роли и безопасность. | Можно упростить архитектуру. |
| Нужна ли работа без интернета? | Нужен кэш или локальная база. | Можно получать данные напрямую с API. |
| Есть ли админская часть? | Нужно описать backend и роли. | Можно оставить только мобильное приложение. |
| Проект делает один студент? | Лучше простая архитектура. | Можно выделить больше модулей и интеграций. |

## **13. Работа с ИИ**

Нейросеть можно использовать как помощника архитектора, но нельзя просто скопировать ответ. Студент должен проверять, не предлагает ли ИИ слишком сложную архитектуру для простого MVP.

Промт 1. Помоги выбрать архитектуру:

«Я делаю мобильное приложение [описание]. Есть такие функции: [список]. Данные: [что хранится]. Нужна/не нужна авторизация. Нужен/не нужен backend. Предложи 2-3 варианта архитектуры, сравни их и выбери оптимальный для студенческого MVP. Не усложняй решение микросервисами без необходимости».

Промт 2. Помоги выделить модули:

«На основе моего ТЗ, User Flow и ERD выдели модули мобильного приложения. Для каждого модуля укажи ответственность, входные данные, выходные данные и связи с другими модулями».

Промт 3. Проверь архитектуру:

«Проверь мою архитектуру приложения. Найди слабые места: лишние зависимости, отсутствие обработки ошибок, нарушение слоев, неясные связи между модулями. Предложи исправления простым языком».

Важно: итоговая архитектура должна быть адаптирована студентом под свой проект, а не оставаться общим шаблоном из нейросети.

## **14. Как оформить в Notion / Buildin**

В документации проекта должна появиться отдельная страница «Архитектура приложения». Она нужна, чтобы преподаватель мог быстро понять техническое устройство проекта.

| Блок страницы | Что вставить |
|---|---|
| 1. Краткое описание архитектуры | 2-3 абзаца: какой вариант выбран и почему. |
| 2. Системный контекст | Картинка схемы + ссылка на редактируемый исходник. |
| 3. Контейнерная схема | Мобильное приложение, API/backend, БД, внешние сервисы. |
| 4. Модули приложения | Схема модулей и таблица ответственности. |
| 5. Потоки данных | 1-2 схемы ключевых сценариев. |
| 6. Структура проекта | Папки, классы, назначение файлов. |
| 7. ADR | 2-3 архитектурных решения. |
| 8. Итоговый вывод | Что архитектура упрощает, а какие риски остаются. |

Схемы вставляются картинками для просмотра, но обязательно должна быть ссылка на редактируемый вариант: FigJam, diagrams.net, Miro, Excalidraw, Mermaid, PlantUML или другой инструмент.

## **15. Инструменты для схем**

| Инструмент | Для чего использовать | Что сдавать |
|---|---|---|
| FigJam / Figma | Красивые схемы контекста, модулей, user flow и потоков данных. | Ссылка на файл + PNG в документации. |
| diagrams.net / draw.io | Строгие технические схемы, контейнеры, компоненты, связи. | Файл .drawio или ссылка + PNG. |
| Excalidraw | Быстрые наброски архитектуры в ручном стиле. | Ссылка/файл + PNG. |
| Mermaid | Текстовые схемы, которые легко хранить в GitHub/Notion. | Код Mermaid + картинка. |
| PlantUML | UML-диаграммы компонентов, классов и последовательностей. | Код PlantUML + картинка. |
| Miro / Whimsical | Визуальные схемы и совместная работа. | Ссылка + PNG. |

Требование: схема не должна быть только скриншотом. У преподавателя должна быть возможность открыть исходник и увидеть, что схема редактируемая.

## **16. Практическое задание**

Задание: на основе своего итогового мобильного проекта спроектировать архитектуру приложения и оформить ее в Notion / Buildin.

**1. **Откройте свои материалы из предыдущих частей: ТЗ, требования, User Flow, дизайн экранов, ERD.

**2. **Определите, какой архитектурный вариант подходит проекту: локальное приложение, клиент-сервер, гибрид или другой вариант.

**3. **Напишите краткое описание выбранной архитектуры: 5-7 предложений.

**4. **Создайте схему системного контекста: пользователь, мобильное приложение, backend/API, база данных, внешние сервисы.

**5. **Создайте контейнерную схему: из каких крупных частей состоит система.

**6. **Создайте схему модулей мобильного приложения.

**7. **Создайте 1-2 схемы потоков данных для ключевых сценариев.

**8. **Опишите структуру папок и основных классов/файлов проекта.

**9. **Оформите 2-3 ADR по важным решениям.

**10. **Вставьте все материалы в Notion / Buildin и приложите ссылки на редактируемые схемы.

## **20. Частые ошибки**

| Ошибка | Как исправить |
