# **Что такое требование**

### Требование — это зафиксированное описание того, что система должна делать или каким условиям должна соответствовать. Хорошее требование помогает не спорить на уровне «мне казалось» и «я думал, что это очевидно», а проверять проект по конкретным пунктам.

### В проекте требование должно быть достаточно простым, но не расплывчатым. Оно должно отвечать на вопрос: что именно должно быть реализовано и как понять, что это действительно работает.

| Слабая формулировка | Почему плохо | Нормальная формулировка |
|---|---|---|
| Сделать удобное приложение. | Непонятно, что именно считается удобным. | Пользователь должен видеть главный экран с основными разделами приложения после запуска. |
| Добавить профиль. | Непонятно, какие данные есть в профиле и что с ними можно делать. | Пользователь может открыть экран профиля и посмотреть имя, аватар, краткое описание и список своих действий. |
| Сделать безопасность. | Слишком общая фраза, по ней нельзя проверить работу. | Пароль пользователя не должен отображаться открытым текстом при вводе. |
| Приложение должно быстро работать. | Нет измеримого критерия. | Главный экран должен открываться не дольше 2 секунд при стабильном интернет-соединении или при локальных данных. |

# **Два основных типа требований**

### В этой части нас интересуют два основных типа: функциональные и нефункциональные требования. Их нельзя смешивать в одну кучу, потому что они отвечают на разные вопросы.

| Тип требования | На какой вопрос отвечает | Пример |
|---|---|---|
| Функциональное требование | Что система должна делать? | Пользователь может создать новую заметку с названием и текстом. |
| Нефункциональное требование | Как система должна работать? | Список заметок должен открываться без заметной задержки при количестве до 100 записей. |
| Ограничение | В каких рамках выполняется проект? | Проект разрабатывается только под Android и использует локальное хранение данных. |
| Критерий приемки | Как проверить, что требование выполнено? | После сохранения заметка появляется в общем списке и не исчезает после перезапуска приложения. |

# **Функциональные требования**

### Функциональные требования **описывают действия**, которые приложение должно позволять выполнять пользователю, администратору или самой системе. Проще говоря, это список возможностей продукта.

### Формула функционального требования: кто выполняет действие, что он делает, с каким объектом работает и какой результат должен получить.

### **Примеры функциональных требований**

| ID | Функциональное требование | Пояснение |
|---|---|---|
| FR-01 | Пользователь может пройти регистрацию, указав имя, email и пароль. | Требование описывает конкретное действие пользователя. |
| FR-02 | Пользователь может войти в приложение по email и паролю. | Авторизация — отдельная функция, ее нельзя прятать внутри регистрации. |
| FR-03 | Пользователь может добавить новую задачу с названием, описанием и сроком выполнения. | Понятно, какие поля нужны для создания объекта. |
| FR-04 | Пользователь может изменить статус задачи: «запланировано», «в работе», «выполнено». | Есть конкретные допустимые состояния. |
| FR-05 | Система показывает список задач, отсортированный по ближайшему сроку. | Здесь действие выполняет система, но оно тоже функциональное. |

## **Как писать функциональные требования**

### • Начинайте с роли: пользователь, студент, преподаватель, администратор, система.

### • Используйте проверяемый глагол: создать, открыть, изменить, удалить, найти, отправить, сохранить, показать.

### • Указывайте объект действия: задача, профиль, курс, заявка, запись, комментарий, уведомление.

### • Добавляйте важные условия: обязательные поля, статусы, ограничения, порядок отображения.

### • Не пишите «сделать красиво», «добавить нормальный экран», «чтобы было удобно» — это не функциональное требование.

| Плохой глагол | Почему не подходит | Лучше использовать |
|---|---|---|
| Улучшить | Непонятно, что именно изменится. | Добавить / изменить / сократить / показать |
| Сделать красиво | Нельзя объективно проверить. | Оформить экран по прототипу / использовать единые отступы и шрифты |
| Оптимизировать | Нет конкретики. | Сократить время загрузки / уменьшить количество запросов / сохранить данные локально |
| Реализовать всё | Нет границ и приоритета. | Реализовать конкретный сценарий MVP |

# **Нефункциональные требования**

### Нефункциональные требования описывают не сами функции, а качество работы приложения. Они отвечают на вопрос: каким должно быть приложение, когда оно выполняет свои функции.

### Если функциональное требование говорит «пользователь может открыть список задач», то нефункциональное уточняет: список должен открываться быстро, корректно отображаться на разных экранах, не ломаться при пустых данных и не показывать лишнюю личную информацию.

| Группа | Что описывает | Пример для мобильного проекта |
|---|---|---|
| Производительность | Скорость открытия экранов, обработки действий, загрузки данных. | Главный экран должен открываться не дольше 2 секунд при нормальных условиях. |
| Удобство | Понятность интерфейса, читаемость, логичность навигации. | Основное действие на экране должно быть видно без прокрутки. |
| Надежность | Поведение приложения при ошибках, пустых данных, перезапуске. | Созданные пользователем записи не должны исчезать после перезапуска приложения. |
| Безопасность | Защита данных, паролей, доступа к личной информации. | Пароль не должен храниться и отображаться в открытом виде. |
| Совместимость | Работа на нужных устройствах и версиях платформы. | Интерфейс должен корректно отображаться на экранах смартфонов от 5.5 до 7 дюймов. |
| Сопровождаемость | Понятность структуры проекта для доработки. | Код должен быть разделен на экраны, модели данных и логику работы с хранилищем. |
| Доступность | Комфорт для пользователей с разными условиями восприятия. | Текстовые элементы должны быть читаемыми, а важные действия не должны зависеть только от цвета. |

## **Как не путать функциональные и нефункциональные требования**

### Если требование можно превратить в кнопку, экран, действие или сценарий — чаще всего оно функциональное. Если оно описывает качество, ограничение, скорость, безопасность или удобство этой функции — чаще всего оно нефункциональное.

| Фраза | Тип | Почему |
|---|---|---|
| Пользователь может создать карточку привычки. | Функциональное | Описано действие пользователя. |
| Карточка привычки должна сохраняться после перезапуска приложения. | Нефункциональное | Описано качество надежности хранения. |
| Пользователь может найти курс по названию. | Функциональное | Описано действие поиска. |
| Поиск должен выдавать результат не дольше 1 секунды при списке до 200 элементов. | Нефункциональное | Описано требование к скорости. |
| Проект выполняется одним студентом за ограниченный срок. | Ограничение | Описана рамка проекта, а не функция приложения. |

# **Ограничения проекта**

### Ограничения — это рамки, в которых выполняется проект. Они не всегда являются функциями, но они сильно влияют на требования. Если не зафиксировать ограничения, проект легко превратится в бесконечный список желаний.

| Тип ограничения | Пример |
|---|---|
| Сроки | Проект должен быть подготовлен к защите до установленной даты. |
| Платформа | Приложение разрабатывается только под Android. |
| Технологии | Для хранения данных используется локальная база или заранее подготовленные данные. |
| Команда | Проект выполняется индивидуально или малой учебной группой. |
| Объем MVP | В первую версию не входят чат, платежи, сложная аналитика и полноценная админ-панель. |
| Данные | Используются тестовые данные, если нет готового сервера или внешнего API. |

# **Критерии приемки**

### Критерии приемки показывают, как проверить, что требование действительно выполнено. Это мост между формулировкой требования и проверкой проекта. Без критериев приемки требование часто остается красивой фразой.

### Критерий приемки должен быть конкретным: что нужно сделать, что должно произойти, какое сообщение появится, какие данные сохранятся, какой экран откроется.

| Требование | Критерии приемки |
|---|---|
| Пользователь может создать задачу. | 1) Поле названия обязательно. 2) После нажатия «Сохранить» задача появляется в списке. 3) После перезапуска приложения задача остается в списке. |
| Пользователь может войти в приложение. | 1) При верных данных открывается главный экран. 2) При пустых полях появляется сообщение об ошибке. 3) Пароль скрывается при вводе. |
| Пользователь может открыть профиль. | 1) Экран профиля открывается из нижнего меню. 2) На экране отображаются имя, аватар и краткая информация. 3) Если данных нет, показывается понятная заглушка. |

# **Приоритеты требований**

### Не все требования одинаково важны. Для проекта удобно использовать простую классификацию Must Have, Should Have, Could Have, Won’t Have. Она помогает связать требования с MVP и не перегружать первую версию.

| Приоритет | Что означает | Как использовать в проекте |
|---|---|---|
| Must Have | Обязательно должно быть в MVP. | Без этого основной сценарий проекта не работает. |
| Should Have | Желательно реализовать, если хватает времени. | Улучшает продукт, но не является ядром MVP. |
| Could Have | Можно добавить как дополнительную возможность. | Подходит для расширения после основной реализации. |
| Won’t Have | Не входит в текущую версию. | Фиксирует, что функция осознанно отложена. |

# **Как получить требования из MVP**

### Требования не нужно придумывать заново. Они должны вытекать из вашего MVP, анкетирования, целевой аудитории и выбранной модели разработки. Если требование никак не связано с проблемой пользователя или главным сценарием, его стоит отложить.

### 1. Возьмите главный пользовательский сценарий из части про MVP.

### 2. Разбейте сценарий на действия: открыть, выбрать, создать, сохранить, посмотреть результат.

### 3. Для каждого действия сформулируйте функциональное требование.

### 4. Для важных действий добавьте нефункциональные требования: скорость, надежность, безопасность, удобство.

### 5. Проверьте, какие требования обязательны для MVP, а какие можно перенести в будущие версии.

### 6. Для ключевых требований напишите критерии приемки.

### В конце страницы напишите короткий вывод на 5–8 предложений. В выводе нужно объяснить, какие требования являются ядром MVP, какие требования отложены на будущие версии и как выбранная модель разработки поможет работать с этими требованиями.

# **Можно использовать нейросеть как помощника**

### Нейросеть можно использовать для проверки формулировок, поиска расплывчатых требований и приведения таблицы к единому стилю. Но итоговые требования должны быть связаны с вашим проектом, вашей аудиторией, вашим MVP и вашей логикой разработки.

### Не стоит просить нейросеть «придумай требования к любому приложению». Лучше дать ей ваш MVP, описание аудитории и список функций, а потом попросить найти слабые формулировки и предложить более проверяемые варианты.

| Можно | Нельзя |
|---|---|
| Попросить проверить, какие требования слишком общие. | Скопировать общий список требований без связи с проектом. |
| Попросить разделить функции на FR и NFR. | Выдать нейросетевой текст за анализ своего MVP. |
| Попросить улучшить критерии приемки. | Добавить требования, которые невозможно реализовать или показать. |
| Попросить найти противоречия между MVP и требованиями. | Оставить требования без проверки и объяснения. |

# **Что нужно сдать**

### • Ссылку на страницу «Требования проекта» в Notion / Buildin.

### • Таблицу функциональных требований: минимум 8 требований.

### • Таблицу нефункциональных требований: минимум 5 требований.

### • Список ограничений проекта.

### • Критерии приемки для требований Must Have.

### • Короткий вывод о связи требований с MVP и выбранной моделью разработки.

### • Аккуратное оформление: заголовки, таблицы, понятные формулировки, отсутствие случайных функций.

# **На что я буду смотреть при проверке**

| Критерий | Что считается хорошим результатом |
|---|---|
| Связь с MVP | Требования помогают реализовать первую версию продукта, а не расписывают абстрактную идеальную систему. |
| Понятность формулировок | По каждому требованию понятно, кто что делает и какой результат должен получиться. |
| Разделение FR и NFR | Функции не смешаны с качеством работы, ограничениями и критериями приемки. |
| Проверяемость | Требования можно проверить на защите или во время демонстрации проекта. |
| Реалистичность | Объем требований соответствует учебному проекту, срокам и возможностям команды. |
| Оформление | Страница оформлена аккуратно: таблицы читаются, заголовки есть, вывод написан своими словами. |

# **Типичные ошибки**

### • Писать требования слишком общо: «сделать удобно», «добавить безопасность», «реализовать красивый интерфейс».

### • Включать в MVP слишком много функций, которые не нужны для проверки главной идеи.

### • Смешивать функциональные требования, нефункциональные требования, ограничения и критерии приемки в одной таблице без структуры.

### • Не указывать приоритеты, из-за чего непонятно, что обязательно делать первым.

### • Не писать критерии приемки, из-за чего невозможно проверить готовность работы.

### • Копировать универсальные требования из интернета без связи с собственным проектом.
