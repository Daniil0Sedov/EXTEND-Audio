# **Зачем проектировать базу данных**

#### База данных — это не место, куда просто «складывается информация». Это основа приложения, от которой зависят регистрация, авторизация, сохранение данных, история действий, статистика, поиск, фильтрация и корректная работа интерфейса.

#### Если база данных спроектирована плохо, приложение быстро сталкивается с проблемами: данные повторяются, связи ломаются, сложно добавить новый экран, невозможно нормально фильтровать записи, появляются ошибки при удалении и обновлении данных.

#### Проектирование базы данных нужно для того, чтобы заранее понять: какие данные есть в системе, как они связаны, что должно храниться обязательно, какие данные нельзя дублировать и какие правила целостности нужны.

# **Что такое сущность, атрибут и запись**

#### Перед ERD и SQL нужно научиться видеть данные в проекте. Для этого используются три базовых понятия: сущность, атрибут и запись.

| Понятие | Что означает | Пример |
|---|---|---|
| Сущность | Объект предметной области, о котором нужно хранить данные | Пользователь, курс, урок, заказ, товар |
| Атрибут | Характеристика сущности | имя пользователя, email, название курса |
| Запись | Один конкретный экземпляр сущности | Иван Иванов, курс «Python», урок «Списки» |

#### Если в приложении есть экран профиля, скорее всего появится сущность Пользователь или Профиль. Если есть список учебных модулей, появится сущность Модуль или Курс. Если есть прогресс прохождения, нужна отдельная сущность Прогресс.

| Простой прием Чтобы найти сущности, выпишите существительные из описания проекта: пользователь, курс, урок, заявка, заказ, сообщение, категория, комментарий, результат. Затем проверьте, какие из них реально нужно хранить в базе. |
|---|

# **Ассоциация и декомпозиция**

#### Ассоциация — это связь между сущностями. Например, пользователь создает задачу, курс содержит уроки, студент проходит модуль, заказ включает товары. Ассоциация помогает понять, какие таблицы должны быть соединены между собой.

#### Декомпозиция — это разбиение большой идеи на меньшие понятные части. В базе данных декомпозиция помогает перейти от фразы «сделать приложение для обучения» к конкретным сущностям: пользователь, курс, урок, прогресс, тест, результат.

#### 

#### *Схема 2. Декомпозиция идеи приложения до сущностей базы данных*

| Что анализируем | Что ищем |
|---|---|
| Описание проекта | Основные объекты, которые нужно хранить |
| Функциональные требования | Данные, которые нужны для выполнения функций |
| User Flow | Данные, которые появляются на каждом шаге пользователя |
| Экраны приложения | Блоки, поля ввода, списки, карточки |
| MVP | Минимальный набор сущностей для первой версии |

# **Типы связей между сущностями**

#### После выделения сущностей нужно определить связи. Связь показывает, как записи одной таблицы относятся к записям другой таблицы.

#### 

#### *Схема 3. Основные типы связей между сущностями*

| Тип связи | Как читать | Пример | Что важно |
|---|---|---|---|
| 1 : 1 | Одна запись связана только с одной записью | Пользователь — профиль | Часто можно объединить в одну таблицу, если нет причины разделять |
| 1 : N | Одна запись связана со многими | Курс — уроки | Самый частый тип связи |
| M : N | Многие записи связаны со многими | Студенты — курсы | Обычно нужна промежуточная таблица |

| Частая ошибка Студенты часто пытаются хранить связь M:N прямо в одной таблице через список значений. Так делать не нужно. Для связи «многие ко многим» обычно создается отдельная таблица-связка. |
|---|

# **ERD-диаграмма**

#### ERD — это диаграмма сущностей и связей. Она нужна, чтобы визуально показать структуру базы данных: какие таблицы есть, какие поля в них находятся, где первичные и внешние ключи, как таблицы связаны между собой.

#### 

#### *Схема 4. Пример ERD для учебного мобильного приложения*

| Элемент ERD | Что означает |
|---|---|
| Таблица / сущность | Объект, данные о котором хранятся в базе |
| Поле / атрибут | Характеристика сущности |
| PK | Primary Key, уникальный идентификатор записи |
| FK | Foreign Key, ссылка на запись в другой таблице |
| Связь | Линия между таблицами, показывающая зависимость |
| Кардинальность | Тип связи: 1:1, 1:N, M:N |

#### ERD должна быть понятной без устного объяснения. Если на схеме нет ключей, типов связей и названий таблиц, она не считается полноценной схемой базы данных.

# **UML-диаграмма моделей данных**

#### UML-диаграмма классов может использоваться рядом с ERD. Если ERD ближе к базе данных и таблицам, то UML ближе к программным моделям приложения: классам, полям, методам и связям между объектами.

#### 

#### *Схема 5. Пример UML-диаграммы моделей данных*

| ERD | UML |
|---|---|
| Показывает структуру базы данных | Показывает модели в коде и их связи |
| Таблицы, поля, PK, FK | Классы, свойства, методы |
| Нужна для проектирования БД | Нужна для понимания логики приложения |
| Ближе к SQL | Ближе к ООП и архитектуре приложения |

| Как использовать в итоговом проекте Для этой части достаточно сделать ERD как основную схему базы данных и UML-диаграмму основных моделей, чтобы показать связь между проектированием базы и будущей реализацией в коде. |
|---|

# **Нормализация базы данных**

#### Нормализация — это процесс приведения структуры данных к более аккуратному виду. Она нужна, чтобы убрать лишнее дублирование, разделить разные типы данных по таблицам и снизить риск ошибок при обновлении, удалении и добавлении записей.

#### 

#### *Схема 6. Пример нормализации данных*

| Нормальная форма | Простое объяснение | Проверочный вопрос |
|---|---|---|
| 1НФ | В ячейке хранится одно значение, нет списков внутри поля | Нет ли в одном поле нескольких значений через запятую? |
| 2НФ | Неключевые поля зависят от всего ключа, а не от его части | Не хранится ли в таблице лишняя информация, относящаяся только к части составного ключа? |
| 3НФ | Поля зависят от ключа, а не друг от друга | Нет ли поля, которое можно вычислить или получить из другой таблицы? |

| Не нужно усложнять Для учебного итогового проекта достаточно уверенно объяснить 1НФ, 2НФ и 3НФ на своей структуре. Главное — показать, что студент осознанно убрал дублирование и разделил данные по смыслу. |
|---|

# **Ключи, ограничения и целостность данных**

#### Ключи и ограничения нужны, чтобы база данных не превратилась в набор случайных таблиц. Они помогают сохранять целостность данных: не допускать повторов, пустых обязательных полей, ссылок на несуществующие записи и некорректных значений.

| Элемент | Назначение | Пример |
|---|---|---|
| Primary Key | Уникально определяет запись | users.id |
| Foreign Key | Связывает таблицу с другой таблицей |  |
| UNIQUE | Запрещает повторы | email должен быть уникальным |
| NOT NULL | Поле обязательно для заполнения | title не может быть пустым |
| CHECK | Проверяет допустимые значения | status IN (new, done, failed) |
| DEFAULT | Значение по умолчанию | created_at = текущая дата |

#### При описании базы данных студент должен указать не только названия таблиц и полей, но и правила: какие поля обязательные, какие уникальные, где используются внешние ключи, какие значения допустимы.

# **Архитектуры работы с данными**

#### Архитектура хранения данных зависит от типа приложения. Одни проекты могут работать только с локальной базой данных на устройстве. Другим нужен сервер, авторизация и синхронизация. Третьи используют смешанный подход: часть данных хранится локально, часть приходит с API.

#### 

#### *Схема 7. Возможные архитектуры хранения данных*

| Вариант | Когда подходит | Пример |
|---|---|---|
| Локальная база | Данные нужны только на одном устройстве | заметки, трекер привычек, список дел |
| Клиент-сервер | Нужна авторизация, общие данные, синхронизация | образовательная платформа, маркетплейс, сервис заявок |
| Гибрид | Нужен офлайн-режим и последующая синхронизация | учебное приложение, дневник, мобильный CRM |

| Для мобильного проекта Если проект учебный и выполняется без сервера, можно спроектировать локальную БД. Но в документации нужно честно указать, какие данные в реальном продукте должны храниться на сервере. |
|---|

# **Словарь данных**

#### Словарь данных — это таблица, где подробно описываются все таблицы и поля базы данных. Он нужен, чтобы другой человек мог понять структуру базы без чтения кода.

| Таблица | Поле | Тип | Ключ | Обязательно | Описание |
|---|---|---|---|---|---|
| users | id | UUID / INTEGER | PK | Да | Уникальный идентификатор пользователя |
| users | email | VARCHAR | UNIQUE | Да | Email для входа в систему |
| lessons | course_id | UUID / INTEGER | FK | Да | Ссылка на курс |
| progress | status | VARCHAR | — | Да | Статус прохождения урока |

#### В итоговой работе словарь данных обязателен. Он показывает, что студент не просто нарисовал ERD, а продумал поля, типы данных, ключи и назначение каждого элемента.

| Минимальный состав словаря Для каждой таблицы укажите: название поля, тип данных, является ли поле ключом, обязательность, ограничение и краткое описание. |
|---|

# **CRUD-матрица**

#### CRUD-матрица показывает, какие операции выполняются с каждой сущностью: создание, чтение, обновление и удаление. Она помогает связать базу данных с пользовательскими сценариями и функциональными требованиями.

| Сущность | Create | Read | Update | Delete | Где используется |
|---|---|---|---|---|---|
| User | регистрация | профиль | редактирование профиля | удаление аккаунта | экран профиля |
| Course | админ создает курс | каталог курсов | редактирование курса | архивация курса | экран каталога |
| Lesson | добавление урока | просмотр урока | обновление урока | удаление урока | экран урока |
| Progress | создание прогресса | просмотр прогресса | изменение статуса | сброс прогресса | личный кабинет |

#### Если у сущности нет ни одного пользовательского сценария и ни одной операции, возможно, она лишняя для MVP. Если сценарий есть, но сущности нет, значит структура базы неполная.

# **Индексы, производительность и безопасность**

#### Даже в учебном проекте важно показать базовое понимание производительности и безопасности данных. Не нужно подробно рассчитывать нагрузку, но нужно объяснить, какие поля будут часто использоваться для поиска, фильтрации и связей.

| Тема | Что нужно указать в работе |
|---|---|
| Индексы | Какие поля часто используются для поиска: email, category_id, course_id, created_at |
| Пароли | Пароль не хранится открытым текстом, используется хеш |
| Персональные данные | Не хранить лишние данные, которые не нужны приложению |
| Удаление | Продумать, что происходит при удалении пользователя или связанной записи |
| Резервное копирование | Для серверного варианта указать необходимость бэкапов |
| Ограничение доступа | Пользователь видит только свои данные, если это требуется логикой приложения |

| Важно Индексы не нужно ставить на каждое поле. Их используют там, где данные часто ищут, фильтруют, сортируют или связывают через FK. |
|---|
