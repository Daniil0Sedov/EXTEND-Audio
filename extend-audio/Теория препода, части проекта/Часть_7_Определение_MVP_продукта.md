### **Что такое MVP**

### MVP расшифровывается как Minimum Viable Product — минимально жизнеспособный продукт. В учебном проекте MVP можно понимать как первую рабочую версию приложения, которая **решает одну главную проблему ****одной конкретной аудитории** и позволяет получить обратную связь.

| Слово | Что означает для проекта |
|---|---|
| Minimum — минимальный | В MVP входит только то, без чего невозможно проверить основную идею продукта. Все дополнительные украшения, редкие сценарии и сложные функции переносятся на следующие версии. |
| Viable — жизнеспособный | Продукт должен быть не просто макетом. Пользователь должен понимать, какую пользу он получает, и иметь возможность пройти основной сценарий. |
| Product — продукт | Даже если версия небольшая, у нее должны быть понятная аудитория, проблема, ценность, основной сценарий, результат и критерии проверки. |

| Важно MVP не обязан включать все идеи команды. Его задача — проверить самое важное: действительно ли выбранная аудитория сталкивается с проблемой и помогает ли предложенное решение. |
|---|

### **Как обычно создают MVP**

### MVP создают не с выбора красивых экранов и не с полного списка функций. Обычно работа начинается с проблемы пользователя, затем формулируется гипотеза решения, после этого выбирается главный сценарий и только потом определяется минимальный набор функций.

| Шаг | Что делается |
|---|---|
| 1. Определяют аудиторию | Кто конкретно будет пользоваться продуктом: школьники, студенты, преподаватели, начинающие разработчики, спортсмены, туристы, жители города и т.д. |
| 2. Формулируют проблему | Что у пользователя болит, мешает, занимает много времени, вызывает ошибки или неудобства. |
| 3. Проверяют гипотезу | Через анкетирование, интервью, наблюдение, анализ аналогов или обсуждение с потенциальными пользователями. |
| 4. Выделяют главный сценарий | Одно основное действие пользователя, ради которого создается первая версия: добавить задачу, найти место, записаться, получить маршрут, сохранить заметку. |
| 5. Отбирают функции MVP | Оставляют только must-have функции. Всё, что можно убрать без разрушения главной пользы, переносится на следующие версии. |
| 6. Проектируют простую структуру | Описывают экраны, данные, роли, переходы, состояния и минимальный пользовательский путь. |
| 7. Собирают первую версию | Создают прототип, рабочую мобильную версию или набор экранов с логикой, достаточной для проверки идеи. |
| 8. Проверяют на пользователях | Показывают MVP аудитории, собирают замечания, фиксируют выводы и планируют следующую итерацию. |

### **На основе чего строится MVP**

### MVP не придумывается отдельно от предыдущих частей проекта. Он должен опираться **на уже собранные материалы**. Если MVP не связан с проблемой, анкетированием и целевой аудиторией, он превращается в случайный набор функций.

| Основа MVP | Как используется |
|---|---|
| Проблема | Какую конкретную проблему решает продукт. Без проблемы MVP теряет смысл. |
| Целевая аудитория | Для кого делается первая версия. MVP не обязан сразу подходить всем пользователям. |
| Гипотеза | Что команда предполагает: «если мы сделаем такую функцию, пользователю станет проще…». |
| Анкетирование / обратная связь | Какие ответы подтвердили или уточнили проблему, функции, боли и ожидания пользователей. |
| Аналоги | Какие решения уже существуют, что в них удобно, что можно улучшить, чего не хватает. |
| Главный пользовательский сценарий | Что пользователь должен сделать в приложении от начала до результата. |
| Ограничения проекта | Время, уровень навыков команды, доступные технологии, учебный формат, требования преподавателя. |
| Критерии успеха | Как команда поймет, что MVP получился: пользователь прошел сценарий, понял ценность, оставил обратную связь, нашел решение своей задачи. |

### **Что должно быть в MVP**

### В MVP должны входить не все возможные функции, а только те, которые доказывают главную ценность продукта. Для этого удобно делить функции на обязательные, желательные, дополнительные и отложенные.

| Категория | Решение | Как понять |
|---|---|---|
| Must-have | Обязательно входит в MVP | Без этой функции нельзя проверить основную идею продукта. |
| Should-have | Желательно, но не критично | Функция полезна, но MVP может быть показан и без неё. |
| Could-have | Можно добавить позже | Функция улучшает продукт, но не влияет на первую проверку гипотезы. |
| Won’t-have now | Не входит в первую версию | Функция может появиться в будущем, но сейчас она усложняет проект и размывает фокус. |

| Правило для отбора функций Если функцию можно убрать, а пользователь всё равно сможет пройти главный сценарий и понять ценность продукта, значит эта функция не обязательна для MVP. |
|---|

### **Что не должно попадать в MVP без необходимости**

### Сложная система ролей, если в первой версии достаточно одного типа пользователя.

### Полноценная админ-панель, если данные можно подготовить заранее или внести вручную.

### Чат, рейтинги, достижения, рекомендации, если они не проверяют основную идею.

### Сложная аналитика, если достаточно простого результата или списка действий.

### Много экранов настроек, если пользовательский сценарий можно пройти без них.

### Авторизация через несколько внешних сервисов, если для проверки MVP достаточно простого входа или локального сценария.

### Красивые, но необязательные анимации, кастомизация, темы оформления и второстепенные визуальные элементы.

### Задание 6. Определите критерии готовности и успеха MVP

### Критерии готовности показывают, что MVP можно сдавать и демонстрировать. Критерии успеха показывают, что MVP не просто существует, а действительно проверяет идею продукта.

| Тип критерия | Что означает | Пример |
|---|---|---|
| Критерии готовности | Что должно быть сделано технически и документально | Есть главный экран, работает основной сценарий, есть минимум данных, оформлена документация, можно показать результат заказчику. |
| Критерии успеха | Как понять, что продукт полезен | Пользователь понял сценарий без долгого объяснения, смог получить результат, подтвердил пользу, оставил обратную связь, предложил улучшения. |
| Критерии ограничения | Что осознанно не делается сейчас | Нет сложной авторизации, нет админ-панели, нет полной аналитики, нет всех ролей, потому что они не нужны для первой проверки. |

### **Какие бывают форматы MVP**

### В учебном мобильном проекте MVP может быть разного формата. Главное, чтобы выбранный формат соответствовал уровню проекта и позволял проверить гипотезу.

| Формат MVP | Когда подходит | Пример |
|---|---|---|
| Текстовое описание MVP | Подходит на раннем этапе, когда нужно зафиксировать границы продукта. | Страница в Notion / Buildin с проблемой, сценарием, функциями и критериями. |
| Кликабельный прототип | Подходит, если нужно проверить интерфейс и пользовательский путь. | Figma-прототип экранов приложения без полноценной разработки. |
| Однофункциональное приложение | Подходит, если можно реализовать главную функцию без дополнительных модулей. | Приложение только для добавления и просмотра учебных дедлайнов. |
| MVP на готовых данных | Подходит, если база данных или админка пока усложняют проект. | Список мест, курсов, задач или карточек заранее подготовлен в приложении. |
| Лендинг + анкета | Подходит для проверки интереса к идее до разработки. | Страница с описанием продукта и формой обратной связи. |
| Wizard of Oz | Подходит, если часть работы временно выполняется вручную, а пользователь видит результат как от системы. | Рекомендации подбираются вручную командой, но интерфейс показывает их как результат сервиса. |

| Для этой части Даже если техническая реализация будет позже, сейчас нужно обязательно определить содержательные границы MVP: что входит в первую версию, зачем это нужно и как это будет проверяться. |
|---|

| Пример Проект: мобильное приложение для контроля учебных дедлайнов студентов. MVP продукта — это первая версия приложения для студентов, которая позволяет быстро добавить учебную задачу, указать срок сдачи и увидеть список ближайших дедлайнов. Главная проблема: студенты забывают о небольших заданиях, путают сроки и начинают работу слишком поздно. Гипотеза: если студент сможет быстро заносить задания и видеть ближайшие дедлайны в одном месте, ему будет проще планировать учебную нагрузку и сдавать работы вовремя. В MVP входят: добавление задания, дата дедлайна, список задач, отметка о выполнении, простой экран с ближайшими сроками. В MVP пока не входят: регистрация через внешние сервисы, совместная работа группы, сложная статистика, push-уведомления, календарная синхронизация.  Критерий готовности: пользователь может пройти основной сценарий без объяснения преподавателя. Критерий успеха: после демонстрации минимум несколько студентов подтверждают, что такой сценарий помог бы им не терять учебные задачи. |

### **Можно использовать нейросеть как помощника**

### Нейросеть можно использовать для проверки логики и формулировок. Но итоговая страница должна опираться на ваш проект, вашу аудиторию, вашу анкету и ваши выводы.

| Пример промта Я делаю итоговый мобильный проект на тему [название]. Целевая аудитория: [кто]. Проблема: [какая]. Гипотеза: [какая]. По анкетированию я понял(а), что [вывод]. Помоги определить MVP: главный сценарий, must-have функции, что не входит в первую версию, критерии готовности, критерии успеха и схему MVP для страницы в Notion / Buildin. |
|---|

### **Что нужно сдать**

### Ссылку на страницу «MVP продукта» в Notion / Buildin.

### На странице должно быть краткое определение MVP своими словами.

### Должны быть описаны аудитория, проблема, гипотеза и главная ценность первой версии.

### Должен быть описан основной пользовательский сценарий.

### Должна быть таблица функций: что входит в MVP, что не входит, почему.

### Должна быть схема MVP в виде блоков, стрелок, таблицы или диаграммы.

### Должны быть критерии готовности и критерии успеха.

### Страница должна быть оформлена аккуратно и открываться по ссылке.

### На что я буду смотреть при проверке

| Критерий | Что оценивается |
|---|---|
| Понимание MVP | Студент объясняет MVP не как «обрезанную версию», а как первую проверяемую версию продукта. |
| Связь с проблемой | Функции MVP действительно связаны с заявленной проблемой пользователя. |
| Фокус на аудитории | Понятно, для какой аудитории создается первая версия. |
| Главный сценарий | Есть конкретный путь пользователя от действия до результата. |
| Отбор функций | Функции разделены на обязательные и отложенные, есть объяснение причин. |
| Схема MVP | Схема помогает понять структуру первой версии и проверку гипотезы. |
| Критерии | Есть понятные критерии готовности и успеха MVP. |
| Оформление | Страница в Notion / Buildin читаемая, структурная, без сплошного неразобранного текста. |

### Проще выбрать модель разработки ПО. Модель разработки выбирается не абстрактно, а под конкретный объём первой версии продукта. Если MVP небольшой, но требования будут уточняться после обратной связи, может подойти Agile / Scrum с элементами прототипирования. Если MVP хорошо делится на независимые части, можно использовать инкрементный подход. Если сначала нужно проверить интерфейс, логично начать с прототипной модели. Поэтому сначала фиксируется MVP, а уже затем выбирается модель разработки.
