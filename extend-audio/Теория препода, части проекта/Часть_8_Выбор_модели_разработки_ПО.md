# Какие модели нужно рассмотреть

### Перед выбором ещё раз посмотрите на изученные модели. Не обязательно описывать каждую огромным текстом, но в вашей работе должно быть видно, что вы понимаете разницу между ними.

| Модель | Коротко | Когда особенно подходит |
|---|---|---|
| Каскадная модель (Waterfall) | Последовательные этапы: требования, проектирование, разработка, тестирование, внедрение. | Когда требования заранее понятны, стабильны и почти не будут меняться. |
| V-образная модель | Похожая на Waterfall, но каждому этапу проектирования заранее соответствует уровень тестирования. | Когда важна проверяемость требований, качество и надежность. |
| Итеративная модель | Один и тот же продукт постепенно уточняется через повторяющиеся циклы. | Когда нужно несколько раз дорабатывать решение, пока оно не станет точным. |
| Инкрементная модель | Продукт собирается частями: каждый инкремент добавляет рабочий кусок функциональности. | Когда проект можно разделить на модули и быстро показать MVP. |
| Спиральная модель | Разработка идет витками, на каждом витке отдельно анализируются риски. | Когда проект крупный, дорогой, рискованный и цена ошибки высокая. |
| Agile / Scrum | Работа короткими спринтами, постоянная обратная связь и гибкое изменение плана. | Когда требования могут меняться, а пользователи должны регулярно видеть результат. |
| RAD | Быстрая разработка через готовые компоненты, конструкторы, шаблоны и частые демонстрации. | Когда нужно быстро собрать типовое приложение из готовых решений. |
| Прототипная модель | Сначала создается прототип, чтобы проверить идею, интерфейс и ожидания пользователя. | Когда непонятен UX, сценарий или точный вид будущего продукта. |
| DevOps | Непрерывная разработка, автоматизация сборки, тестирования, публикации и мониторинга. | Когда продукт нужно постоянно обновлять и поддерживать после релиза. |

# Задание 2. Определите критерии выбора модели

### Перед тем как выбрать модель, ответьте на вопросы ниже. Эти ответы помогут не выбрать модель «по настроению», а доказать выбор логически.

| Вопрос | Как влияет на выбор |
|---|---|
| Насколько понятны требования? | Если требования стабильны — возможен Waterfall. Если будут уточняться — лучше Agile, итеративная или прототипная модель. |
| Насколько понятен пользовательский интерфейс? | Если интерфейс нужно проверять на людях — полезна прототипная модель. |
| Можно ли быстро собрать MVP? | Если да — хорошо подходит инкрементная модель или Agile с инкрементами. |
| Высока ли цена ошибки? | Если ошибка может привести к серьёзным последствиям — рассматривают V-Model или спиральную модель. |
| Делится ли продукт на модули? | Если есть независимые части, например профиль, каталог, карта, уведомления, аналитика — подходит инкрементная логика. |
| Нужна ли постоянная обратная связь? | Если нужно часто показывать результат пользователям и преподавателю — подходит Agile / Scrum. |
| Есть ли готовые компоненты? | Если проект можно быстро собрать из шаблонов, конструкторов или готовых библиотек — можно использовать элементы RAD. |
| Планируется ли публикация и поддержка? | Если продукт будет регулярно обновляться, можно добавить элементы DevOps: GitHub, деплой, проверки, документация. |

| Окак Для большинства студенческих мобильных проектов нормальный выбор — Agile / Scrum, инкрементная модель, прототипная модель или их осмысленная комбинация. Но всё равно нужно объяснить, почему это подходит именно вашему проекту. |
|---|

# Задание 3. Выберите основную модель разработки

### В работе нужно указать одну основную модель. Дополнительно можно написать, какие элементы других моделей вы будете использовать. Например: «Основная модель — Agile / Scrum, дополнительно используются прототипирование экранов и инкрементная поставка функционала».

### Не пишите просто: «Я выбрал Agile, потому что он современный». Это слабое объяснение.

### Не выбирайте DevOps как единственную модель только потому, что проект будет опубликован. DevOps больше относится к культуре поставки, автоматизации и поддержки.

### Не выбирайте Waterfall, если сами ещё не понимаете функции приложения и собираетесь менять идею после обратной связи.

### Не выбирайте спиральную модель для небольшого учебного приложения без серьёзных рисков: она может быть слишком тяжёлой.

### Не выбирайте прототипную модель как финальный ответ, если после прототипа вы не объясняете, как дальше будет идти разработка.

| Тип проекта | Что может подойти | Почему |
|---|---|---|
| Учебное приложение с не до конца понятным интерфейсом | Прототипная модель + Agile / Scrum | Сначала проверить экраны и пользовательский сценарий, затем дорабатывать короткими циклами. |
| Приложение, которое легко делится на функции | Инкрементная модель | Можно выпустить MVP, а затем добавлять профиль, уведомления, поиск, статистику и другие модули. |
| Продукт с частой обратной связью от пользователей | Agile / Scrum | После каждого этапа можно показывать работающий результат и корректировать план. |
| Приложение со строгими требованиями к надежности | V-Model или элементы V-Model | Нужно заранее связать требования с тестами и проверками. |
| Типовое приложение на готовых шаблонах | RAD | Главная ценность — быстро собрать рабочий вариант из готовых компонентов. |
| Большой рискованный проект | Спиральная модель | Каждый цикл должен отдельно снижать ключевые риски. Для обычного учебного проекта используется редко. |

# Задание 5. Сделайте схему разработки проекта
