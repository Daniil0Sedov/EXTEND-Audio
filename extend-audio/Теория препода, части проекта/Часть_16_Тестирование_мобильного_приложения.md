## **ЧТО ТАКОЕ ТЕСТИРОВАНИЕ ПРИЛОЖЕНИЯ**

#### *Тестирование — это проверка, что приложение действительно соответствует требованиям, пользовательским сценариям и техническому заданию.*

#### Тестирование нужно не для того, чтобы “поймать студента на ошибке”, а чтобы доказать: приложение можно показать пользователю, заказчику или комиссии. Если тестирование не оформлено, то невозможно понять, какие функции реально проверялись, какие ошибки были найдены и что исправлено.

#### В итоговом проекте тестирование должно быть связано с предыдущими частями: требованиями, техническим заданием, user flow, базой данных, архитектурой и разработкой приложения.

| Главная мысль Если функция есть в MVP и описана в ТЗ, она должна быть проверена. Если функция не проверена, ее нельзя уверенно считать готовой. |
|---|

### **Что считается объектом проверки**

#### **• **экран приложения и его элементы;

#### **• **пользовательский сценарий;

#### **• **форма ввода данных;

#### **• **переход между экранами;

#### **• **сохранение и загрузка данных;

#### **• **работа с локальной базой данных или API;

#### **• **обработка ошибок;

#### **• **соответствие дизайну и требованиям;

#### **• **стабильность приложения при обычных действиях пользователя.

## **ВИДЫ ТЕСТИРОВАНИЯ В ИТОГОВОМ ПРОЕКТЕ**

#### В учебном проекте не нужно пытаться проверить “вообще все”. Нужно выбрать разумный набор проверок, который доказывает работоспособность MVP. Для мобильного приложения достаточно сочетать модульные, интеграционные, UI, ручные и приемочные проверки.

| Вид тестирования | Что проверяет | Пример для мобильного проекта |
|---|---|---|
| Модульное | Одну небольшую часть кода отдельно от остальных | Проверка валидатора email, расчета прогресса, фильтрации списка |
| Интеграционное | Связку нескольких частей приложения | Экран добавления задачи сохраняет данные в БД и отображает их в списке |
| UI / E2E | Путь пользователя через интерфейс | Пользователь входит, открывает каталог, выбирает элемент, сохраняет результат |
| Smoke | Быстрая проверка, что приложение запускается и основные экраны открываются | Запуск приложения, переход на главный экран, открытие профиля |
| Regression | Проверка, что после исправлений старые функции не сломались | После исправления регистрации повторно проверить вход и профиль |
| Приемочное | Соответствие MVP требованиям и критериям готовности | Все Must Have требования из ТЗ выполнены и подтверждены |

## **МОДУЛЬНОЕ ТЕСТИРОВАНИЕ**

#### Модульное тестирование проверяет маленькие части приложения без запуска всего интерфейса. Это самый удобный способ доказать, что внутренняя логика работает правильно.

### **Что можно тестировать модульно**

#### **• **валидацию email, пароля, телефона, имени пользователя;

#### **• **расчет общей суммы, рейтинга, прогресса, процента выполнения;

#### **• **фильтрацию, сортировку и поиск;

#### **• **проверку статуса задачи, заявки, курса или заказа;

#### **• **логику ViewModel / UseCase / Service;

#### **• **преобразование данных из модели в формат для экрана.

| Хороший модульный тест Проверяет одну мысль. Название теста должно объяснять, что проверяется и при каком условии. |
|---|

### **Правило AAA**

| Этап | Смысл | Пример |
|---|---|---|
| Arrange | Подготовить данные | Создать пользователя с корректным email |
| Act | Выполнить действие | Вызвать функцию проверки email |
| Assert | Сравнить результат | Ожидать true или отсутствие ошибки |

### **Пример названий модульных тестов**

#### **• **validEmail_returnsTrue()

#### **• **emptyPassword_returnsValidationError()

#### **• **completedTasks_calculatesProgressCorrectly()

#### **• **searchByTitle_returnsOnlyMatchingItems()

#### **• **negativeAmount_isRejected()

## **ИНТЕГРАЦИОННОЕ ТЕСТИРОВАНИЕ**

#### Интеграционное тестирование проверяет, как несколько частей приложения работают вместе. В мобильном проекте это особенно важно: экран может быть красивым, функция может быть написана, но данные могут не сохраняться, не обновляться или теряться при переходах.

### **Что проверять интеграционно**

#### **• **экран + ViewModel + Repository;

#### **• **экран + локальная база данных;

#### **• **форма + валидация + сохранение результата;

#### **• **авторизация + переход на главный экран;

#### **• **создание записи + отображение в списке;

#### **• **редактирование записи + обновление деталей;

#### **• **удаление записи + отсутствие ее в списке;

#### **• **работа с API или мок-данными, если серверной части нет.

| Интеграционный сценарий | Что связывает | Что должно получиться |
|---|---|---|
| Создание записи |  | Новая запись отображается после сохранения |
| Редактирование профиля |  | Измененные данные сохраняются и отображаются после перезапуска |
| Фильтрация каталога |  | На экране остаются только подходящие элементы |
| Навигация по сценарию |  | Переходы работают без потери состояния |

## **UI, РУЧНОЕ И ПРИЕМОЧНОЕ ТЕСТИРОВАНИЕ**

#### UI-тестирование проверяет поведение приложения с точки зрения пользователя. Даже если внутренний код работает, приложение может быть неудобным: кнопки не нажимаются, текст не читается, экран не помещается, ошибка не объясняется.

### **Что нужно проверить вручную**

#### **• **приложение запускается без ошибки;

#### **• **главный пользовательский сценарий проходится от начала до конца;

#### **• **все обязательные экраны открываются;

#### **• **кнопки и формы работают ожидаемо;

#### **• **пустые поля и некорректные данные обрабатываются;

#### **• **ошибки понятны пользователю;

#### **• **данные сохраняются после возврата на экран или перезапуска;

#### **• **интерфейс не ломается на разных размерах экрана;

#### **• **нет лишних тестовых надписей, заглушек и непонятных элементов.

| Приемочное тестирование В конце студент сверяет результат с ТЗ: все обязательные требования Must Have должны быть реализованы и проверены. Если требование не выполнено, нужно честно указать статус и причину. |
|---|

## **ТЕСТ-КЕЙС: КАК ОФОРМИТЬ ПРОВЕРКУ**

#### Тест-кейс — это пошаговая инструкция проверки. По хорошему тест-кейсу другой человек сможет повторить действия студента и получить тот же результат.

#### 

### **Шаблон таблицы тест-кейсов**

| ID | Название | Предусловие | Шаги | Ожидаемый результат | Факт | Статус |
|---|---|---|---|---|---|---|
| TC-01 |  |  |  |  |  | Passed / Failed / Blocked |
| TC-02 |  |  |  |  |  | Passed / Failed / Blocked |
| TC-03 |  |  |  |  |  | Passed / Failed / Blocked |
| TC-04 |  |  |  |  |  | Passed / Failed / Blocked |
| TC-05 |  |  |  |  |  | Passed / Failed / Blocked |

### **Минимум по тест-кейсам**

#### **• **не менее 10 тест-кейсов на основные сценарии приложения;

#### **• **не менее 3 негативных тестов, где пользователь вводит неправильные данные;

#### **• **не менее 2 тестов на сохранение и отображение данных;

#### **• **не менее 1 теста на навигацию между экранами;

#### **• **каждый тест-кейс должен иметь фактический результат и статус.

## **ТЕСТОВЫЕ ДАННЫЕ И НЕГАТИВНЫЕ СЦЕНАРИИ**

#### Тестовые данные — это заранее подготовленные значения, на которых проверяется приложение. Студент должен проверить не только “идеальный” случай, но и ошибки пользователя.

| Тип данных | Пример | Что проверяем |
|---|---|---|
| Корректные | ivan@mail.ru, пароль 12345678 | Обычный успешный сценарий |
| Пустые | пустой email, пустой заголовок | Запрет отправки пустой формы |
| Граничные | 1 символ, очень длинный текст | Ограничения длины и устойчивость интерфейса |
| Некорректные | email без @, отрицательная сумма | Валидацию и понятное сообщение об ошибке |
| Дубликаты | повторное название команды, существующий email | Проверку уникальности |
| Удаленные/несуществующие | открыть удаленную запись | Поведение при отсутствии данных |

| Правило Хороший тестировщик проверяет не только путь “все правильно”, но и путь “пользователь ошибся”. |
|---|

## **БАГ-РЕПОРТ: КАК ФИКСИРОВАТЬ ОШИБКУ**

#### Баг-репорт — это карточка ошибки. В ней должно быть понятно, что произошло, где произошло, как повторить ошибку и насколько она мешает работе приложения.

#### 

### **Шаблон баг-репорта**

| Поле | Что указать |
|---|---|
| ID ошибки | BUG-01, BUG-02 и так далее |
| Название | Коротко: “Не сохраняется профиль после редактирования” |
| Экран / модуль | Где обнаружена проблема |
| Шаги воспроизведения | Что нужно сделать, чтобы повторить ошибку |
| Ожидаемый результат | Как должно быть по ТЗ |
| Фактический результат | Что произошло на самом деле |
| Серьезность | Critical / Major / Minor / Trivial |
| Доказательство | Скриншот, видео, лог, ссылка на коммит |
| Статус | Open / Fixed / Retest / Closed |

### **Шкала серьезности**

| Уровень | Когда использовать |
|---|---|
| Critical | Приложение падает, невозможно пройти основной сценарий |
| Major | Важная функция не работает, но приложение не падает |
| Minor | Есть ошибка, но можно продолжить работу |
| Trivial | Опечатка, косметическая проблема, мелкое несоответствие |

## **СКРИНШОТЫ, ВИДЕО И ДОКАЗАТЕЛЬСТВА**

#### Результат тестирования должен быть доказуемым. Недостаточно написать “работает”. Нужно приложить скриншот, видео, лог или ссылку на коммит, где видно, что проверка действительно выполнена.

#### 

### **Что приложить к результатам**

#### **• **скриншоты успешных проверок ключевых сценариев;

#### **• **скриншоты найденных ошибок;

#### **• **короткое видео прохождения главного сценария;

#### **• **логи или текст ошибки, если приложение падает;

#### **• **ссылки на коммиты, где ошибка была исправлена;

#### **• **итоговую таблицу со статусами Passed / Failed / Blocked.

### **Правило именования файлов**

#### **• **TC-LOGIN-01_passed.png — успешная проверка входа;

#### **• **BUG-03_profile-save_failed.png — ошибка сохранения профиля;

#### **• **TC-MAIN-FLOW_video.mp4 — видео основного сценария;

#### **• **BUG-02_fix_commit.txt — ссылка на коммит с исправлением.

## **ТЕСТ-ПЛАН ПРОЕКТА**

#### Тест-план — это краткое описание того, что именно студент собирается проверять. Он помогает не забыть важные части приложения и показывает, что тестирование выполнено осознанно.

| Раздел тест-плана | Что написать |
|---|---|
| Цель тестирования | Проверить готовность MVP мобильного приложения к защите |
| Объект тестирования | Какие экраны, модули и функции проверяются |
| Что не тестируется | Функции, которые не входят в MVP или пока не реализованы |
| Виды тестирования | Модульное, интеграционное, UI, smoke, приемочное |
| Тестовые данные | Какие данные используются для проверок |
| Критерии начала | Приложение запускается, основные экраны реализованы |
| Критерии завершения | Все Must Have проверки выполнены, критические ошибки закрыты |
| Риски | Что может помешать завершить тестирование |

| Важно Тест-план не должен быть огромным. Достаточно 1–2 страниц, но он должен быть связан с реальным приложением студента. |
|---|

## **ОФОРМЛЕНИЕ ТЕСТИРОВАНИЯ В NOTION / BUILDIN**

#### Все результаты тестирования нужно оформить в отдельном разделе проекта. Это понадобится для защиты: преподаватель должен видеть не только приложение, но и последовательность проверки.

#### 

### **Минимальная структура страницы**

#### **• **Цель тестирования;

#### **• **Ссылка на актуальную сборку или репозиторий;

#### **• **Тест-план;

#### **• **Таблица тест-кейсов;

#### **• **Чек-лист ручного тестирования;

#### **• **Таблица баг-репортов;

#### **• **Скриншоты и видео-доказательства;

#### **• **Итоговый отчет: что работает, что исправлено, что осталось.

### **Что должно быть видно преподавателю**

#### **• **какие функции проверялись;

#### **• **какие сценарии прошли успешно;

#### **• **какие ошибки были найдены;

#### **• **какие ошибки исправлены;

#### **• **какие проверки связаны с требованиями из ТЗ;

#### **• **какие доказательства приложены к результатам.

## **РАБОТА С ИИ ПРИ ТЕСТИРОВАНИИ**

#### Нейросеть можно использовать как помощника при подготовке тест-кейсов, негативных сценариев, чек-листов и анализа требований. Но ИИ не должен придумывать результаты тестирования: фактический результат студент заполняет только после реальной проверки приложения.

### **Что можно поручить ИИ**

#### **• **сгенерировать список тест-кейсов по ТЗ;

#### **• **предложить негативные сценарии;

#### **• **найти слабые места в требованиях;

#### **• **подготовить чек-лист UI-тестирования;

#### **• **помочь оформить баг-репорт;

#### **• **предложить названия модульных тестов;

#### **• **проверить, не забыты ли важные сценарии.

### **Шаблон промта для ИИ**

| Промт Ты выступаешь как QA-инженер. На основе описания моего мобильного приложения, требований и пользовательских сценариев составь таблицу тест-кейсов. Раздели проверки на: модульные, интеграционные, UI, негативные и приемочные. Для каждого теста укажи ID, название, предусловие, шаги, ожидаемый результат и приоритет. |
|---|

### **Что нельзя делать**

#### **• **нельзя сдавать сгенерированные тест-кейсы без адаптации под свой проект;

#### **• **нельзя писать Passed, если проверка реально не выполнялась;

#### **• **нельзя придумывать скриншоты, логи или исправления;

#### **• **нельзя скрывать критические ошибки перед защитой.

## **ПРАКТИЧЕСКОЕ ЗАДАНИЕ**

#### На основе своего итогового мобильного проекта подготовьте раздел “Тестирование”. Результат должен быть оформлен в Notion / Buildin и подтвержден ссылками, скриншотами или файлами.

### **Что необходимо выполнить**

#### **1. **Составить краткий тест-план проекта.

#### **2. **Подготовить таблицу тест-кейсов для основных сценариев.

#### **3. **Описать не менее 5 модульных тестов или реализовать их в проекте.

#### **4. **Описать не менее 3 интеграционных сценариев.

#### **5. **Составить чек-лист ручного тестирования экранов.

#### **6. **Провести проверки и заполнить фактические результаты.

#### **7. **Найти и оформить минимум 2 баг-репорта. Если ошибок не найдено, искусственно проверьте негативные сценарии и зафиксируйте результат.

#### **8. **Приложить скриншоты, видео или логи к результатам.

#### **9. **Подготовить итоговый отчет о готовности MVP.

### **Минимальный комплект сдачи**

| Артефакт | Формат сдачи | Обязательно |
|---|---|---|
| Тест-план | страница / блок в Notion или Buildin | да |
| Тест-кейсы | таблица | да |
| Модульные тесты | код или таблица описания | да |
| Интеграционные сценарии | таблица или схема | да |
| Чек-лист UI | таблица | да |
| Баг-репорты | таблица + доказательства | да |
| Скриншоты / видео | вложения или ссылки | да |
| Итоговый отчет | короткий вывод | да |

## **ЧАСТЫЕ ОШИБКИ СТУДЕНТОВ**

| Ошибка | Почему плохо | Как исправить |
|---|---|---|
| “Я просто запустил приложение” | Нет доказательств проверки | Оформить тест-кейсы и скриншоты |
| Нет связи с требованиями | Непонятно, что именно проверялось | Связать тесты с ТЗ и MVP |
| Все тесты только позитивные | Не проверены ошибки пользователя | Добавить негативные сценарии |
| Нет фактического результата | Тест нельзя считать выполненным | Заполнить факт и статус |
| Баг описан одной фразой | Невозможно повторить ошибку | Указать шаги, ожидаемый и фактический результат |
| Скриншоты без подписей | Неясно, к какому тесту они относятся | Называть файлы по ID теста или бага |
| ИИ сгенерировал, студент не проверил | Результаты недостоверны | Проверить вручную и адаптировать под проект |
