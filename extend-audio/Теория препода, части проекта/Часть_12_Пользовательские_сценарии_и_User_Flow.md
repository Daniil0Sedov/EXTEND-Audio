| Главная задача части Собрать редактируемую схему движения пользователя по приложению: от первого входа до достижения цели. Схема должна быть создана в сервисе, где ее можно открыть, исправить и доработать. |
|---|

### **Что студент должен сдать**

#### редактируемую схему User Flow: ссылка на доску или исходный файл;

#### экспорт схемы в PNG или PDF для вставки в отчет;

#### легенду схемы: что обозначают фигуры, стрелки, цвета и подписи;

#### описание пользовательских сценариев в Notion / Buildin;

#### таблицу экранов, переходов и условий перехода.

## **Пользовательский сценарий**

#### Пользовательский сценарий — это описание того, как конкретный пользователь достигает цели внутри приложения. Он показывает не только экраны, но и смысл действий: зачем пользователь пришел, что он нажимает, где может ошибиться и каким должен быть результат.

| Простая формула сценария |
|---|

### **Пример**

| Элемент | Пример заполнения |
|---|---|
| Пользователь | Студент, который хочет найти учебный материал |
| Цель | Открыть нужный модуль и начать выполнение задания |
| Стартовая точка | Главный экран приложения |
| Основные шаги |  |
| Возможная проблема | Модуль не найден или у пользователя нет доступа |
| Результат | Материал открыт, пользователь может приступить к работе |

### **Что важно учесть**

#### Сценарий должен описывать реальное действие пользователя, а не просто перечислять экраны.

#### У каждого сценария должна быть понятная цель и конечный результат.

#### Нужно показать не только идеальный путь, но и хотя бы один проблемный вариант: ошибка, пустое состояние, отказ в доступе, неверные данные.

## **User Flow**

#### User Flow — это схема пути пользователя внутри приложения. Она показывает, с какого экрана пользователь начинает, какие действия выполняет, какие решения принимает система и на какой экран он попадает дальше.

### **Зачем нужен User Flow**

#### помогает увидеть логику приложения до начала разработки;

#### показывает, какие экраны действительно нужны проекту;

#### помогает найти лишние экраны, тупиковые переходы и забытые ошибки;

#### позже используется для проектирования навигации, **архитектуры и тестирования**.

*Схема перехода от требований к User Flow и дальнейшей реализации.*

| Главное требование User Flow должен быть редактируемым. Нельзя сдавать только картинку, сделанную нейросетью. Преподаватель должен иметь возможность открыть схему и увидеть, как она устроена. |
|---|

## **Легенда схемы**

#### Легенда — это пояснение к схеме. Она показывает, что означают фигуры, цвета, стрелки и подписи. Без легенды схема может выглядеть красиво, но ее будет сложно читать и проверять.

*Пример легенды. Все обозначения должны быть понятны без устного объяснения студента.*

### **Требование к легенде**

#### каждый тип фигуры должен иметь понятное значение;

#### стрелки должны показывать направление движения пользователя;

#### развилки должны иметь подписи: «да», «нет», «ошибка», «успешно»;

#### цвета используются только если они что-то обозначают.

## **Инструменты для создания редактируемой схемы**

#### Студент может использовать любой инструмент, если итоговую схему можно открыть и отредактировать. Нейросеть можно использовать для генерации структуры, но финальная схема должна быть проверена и доработана вручную.

| Инструмент | Для чего подходит | Что сдать |
|---|---|---|
| FigJam / Figma | User Flow, карта экранов, командная работа | ссылка на файл или доступ к доске |
| Miro | Большие схемы, совместная доска, быстрые правки | ссылка на доску и экспорт PNG/PDF |
| Whimsical | Быстрая генерация flowchart по текстовому промту | ссылка на схему и экспорт |
| diagrams.net / draw.io | Бесплатные схемы без обязательной регистрации | файл .drawio и экспорт PNG/PDF |
| Excalidraw | Черновые схемы, быстрые визуальные наброски | файл .excalidraw и экспорт |
| Mermaid | Текстовое описание схемы в формате кода | исходный код схемы и экспорт изображения |

| Важно Stitch with Google можно использовать для генерации экранов приложения. Для User Flow лучше использовать FigJam, Miro, Whimsical, diagrams.net, Excalidraw, Mermaid или похожий инструмент, где схема остается редактируемой. |
|---|

### **Роль нейросети**

#### помочь выделить основные сценарии из ТЗ;

#### предложить структуру User Flow;

#### сформулировать подписи для экранов, действий и условий;

#### найти пропущенные ошибки и альтернативные пути пользователя.

## **Практическое задание**

#### На основе своего итогового проекта нужно создать редактируемую схему User Flow и оформить ее в Notion / Buildin. Работа выполняется после генерации дизайна экранов, потому что студент уже понимает, какие основные экраны есть в приложении.

### **Порядок выполнения**

#### Выберите 3–5 основных пользовательских сценариев своего приложения.

#### Для каждого сценария опишите цель пользователя и ожидаемый результат.

#### Составьте промт для нейросети: попросите предложить User Flow, развилки, ошибки и пустые состояния.

#### Перенесите результат в редактируемый инструмент: FigJam, Miro, Whimsical, diagrams.net, Excalidraw, Mermaid или аналог.

#### Добавьте легенду: экраны, действия, условия, ошибки, заметки и стрелки.

#### Отметьте основной путь пользователя и хотя бы один альтернативный путь.

#### Проверьте, нет ли тупиков: каждый путь должен приводить к результату, ошибке или возврату назад.

#### Экспортируйте схему картинкой и приложите ссылку или исходный файл редактируемой схемы.

#### Оформите результат в Notion / Buildin по структуре из этой части.

### **Минимальный состав схемы**

#### 8–15 экранов или логических шагов;

#### не менее 3 пользовательских сценариев;

#### не менее 2 условий или развилок;

#### не менее 2 проблемных ситуаций: ошибка, пустое состояние, отсутствие доступа, неверные данные;

#### легенда и подписи переходов.

## **Дополнительные задания и критерии оценки**

### **Дополнительные задания для усиления проекта**

#### Сгенерировать отдельный Error Flow: что происходит при ошибке входа, отсутствии интернета, пустом списке или недоступном разделе.

#### Сгенерировать Empty States: какие экраны показываются, когда нет данных, заявок, избранного, сообщений или результатов поиска.

#### Сгенерировать Role Flow: разные пути для гостя, авторизованного пользователя и администратора, если роли есть в проекте.

#### Сравнить User Flow с дизайном из предыдущей части и найти экраны, которых не хватает.
